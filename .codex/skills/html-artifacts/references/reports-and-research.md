# Reports, Research & Explainers

Use structured, scannable HTML for concept explainers, codebase feature explainers, status reports, and incident reports.

Concept explainers should open with a one-paragraph TL;DR and the core insight, then use a live demo when the topic is spatial or stateful, a quantitative comparison to the naive approach, a nearby glossary, and tabbed code samples where useful.

Feature explainers should identify what the feature does and where it lives, organize phases in collapsible sections, annotate important code, answer recurring questions, and link readers to the next relevant code locations.

Status reports should visually separate shipped, in-flight, and blocked work; keep each item brief; include a useful chart or trend; and isolate concrete asks.

Incident reports should lead with severity, duration, and customer impact, use a real timestamped timeline, place log excerpts at the relevant moments, explain root cause independently, document what worked and failed, and assign follow-up actions to owners with deadlines.
